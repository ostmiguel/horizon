# Консолидация «Обязательств» — согласованная модель (ledger-first)

> Цель: единый, честный учёт долга без дубля и путаницы. Согласовано 2026-07-02.

## 1. Принцип (главное)

**Операции (ledger) = ЕДИНСТВЕННЫЙ источник истины.** Баланс счёта «Обязательства», STS и все метрики считаются из операций. Причина: реальные долговые процессы у всех разные (кредитка с льготным периодом, пропущенный платёж, страховка, досрочка) — автоматикой их не унифицировать, ledger гибко покрывает всё. **Операции не удаляем и не отдаём автоматике.**

**Роли слоёв:**

| Слой | Роль |
|------|------|
| **Операции (ledger)** | Источник истины: баланс «Обязательства», STS, метрики. |
| **Счёт «Обязательства»** | Системный Пассив, автосоздаётся, баланс = ledger. Ручной `initial_balance` убран (по умолч. 0). |
| **Страница «Кредиты»** (кредиты + долги) | Планировщик + модель + сверка. Кредит даёт график → питает прогноз. Показывает «смоделированную сумму долга». Поверх ledger, НЕ перезаписывает его. |

**Как уходит ручной ввод начальной суммы:** при создании кредита/долга предлагаем создать **founding-операцию** «Взял кредит/долг» с выбором счёта зачисления (куда пришли деньги). Ledger получает стартовую сумму сам.

## 2. Сущности и счета

- **Актив-счета** (Карта, Наличные, Резерв) — кэш.
- **Обязательства** (системный Пассив, автосоздаётся, баланс из ledger) — сколько должен.
- **Дебиторка** — сколько должны тебе (зеркало, трогаем минимально).
- **Расход / Доход** — потоковые.
- **loans** (+ поле `kind`): `credit` (график тело/процент для прогноза) | `debt` (без графика). Разница только в наличии графика; баланс обоих ведёт ledger.

## 3. Каталог операций

| # | Событие | Проводка (ledger) | Эффект |
|---|---------|-------------------|--------|
| 1 | Оформить кредит/долг + деньги на счёт | `Обязательства → Тбанк` (founding) | кэш +, долг + |
| 2 | Трата с кредитки (в долг) | `Обязательства → Расход` | долг +, расход |
| 3 | Плановый платёж: тело | `Карта → Обязательства` | кэш −, долг − |
| 4 | Плановый платёж: процент | `Карта → Расход` (Кредиты-процент) | кэш −, расход |
| 5 | Страховка/комиссия по кредиту | `Карта → Расход` | кэш −, расход |
| 6 | Возврат долга / гашение кредитки | `Карта → Обязательства` | кэш −, долг − |
| 7 | Пропущенный платёж (набежал %) | `Карта → Расход` (или доначислить) | расход |
| 8 | Дебиторка: дал в долг / вернули | `Карта → Дебиторка` / `Дебиторка → Карта` | зеркало |

Кредит с графиком: плановые тело+процент подтверждаются (стр. Кредиты + промпт Бюджета в дату) → создают операции 3 и 4 одним действием.

## 4. Сверка (reconciliation) — ключевой плюс

Раз ledger — истина, а страница «Кредиты» — модель, показываем: **сходится ли** ledger-баланс «Обязательства» со смоделированной суммой из кредитов/долгов. Разошлось (забыл внести страховку/процент/трату) → подсветка «проверь». Ловит ошибки, но **не перезаписывает** автоматом. Даёт доверие к цифре без насильственной автоматизации.

## 5. Метрики

- **Чистый капитал** = активы − |баланс Обязательства| (ledger). Долги входят.
- **DSR (кредитная нагрузка)** = ежемесячные платежи кредитов / доход. **Долги без графика НЕ входят** (нет помесячного платежа) — согласовано.
- **«Свободно» / прогноз** — плановые платежи кредитов (график) вычитаются; долги — по факту операций.

## 6. Различие типов

- **Кредит** = обязательство с фиксированным **графиком** (тело/процент) → питает прогноз, входит в DSR.
- **Долг** = обязательство без графика, баланс ведёт ledger (кредитка, займ у физлица). В DSR не входит.

## 7. Фазы реализации

1. **Фундамент:** миграция `loans.kind`; автосидинг системного счёта «Обязательства» (Пассив); убрать ручной `initial_balance` для него; миграция данных (обнулить ручное, оставить ledger от кредитов).
2. **Долги в UI:** «+ Обязательство» (Кредит/Долг), форма долга, founding-операция «куда пришли деньги», примечание на стр. Кредиты.
3. **Сверка:** ledger-баланс vs смоделированная сумма, подсветка расхождений.
4. **Подтверждение платежей кредита:** confirm-флоу тела+процента (стр. Кредиты + промпт Бюджета) = пункт B3 роадмапа.
