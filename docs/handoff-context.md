## 2. Модель данных

- **Операция = движение денег между двумя счетами:** `account_from → account_to`,
  `amount` всегда положительный, `date`, `category_id`. Кассовый метод.
- «Технические» счета-маркеры (глобальные константы модели, не данные юзера):
  `Доход`, `Расход`, `Обязательства`, `Дебиторка`.
- Реальные счета (`accounts`, per-user): тип `Актив` / `Пассив`; флаги
  `include_in_balance` (в операционном балансе), `is_reserve` / `is_cushion`.
- Таблицы фактов/плана: `transactions` (факт) и `plan` (план) — одинаковая форма,
  один эндпоинт `/api/transactions` (`?plan=true` → таблица `plan`).
- `categories` (**per-user**, `WHERE user_id`): `group_name` (Доходы/Расходы/
  Обязательства/Дебиторка), `category`, `subcategory`, `character`, `expense_type`,
  **`role`** (см. §7).
- `plan_rules` (правила), `category_budgets` (конверты, per category+year+month),
  `goals` (цели), `loans` + `loan_schedule`, `forecast_snapshots`,
  `plan_rule_skips` (см. §6), `sessions`, `users`.

### ⚠️ Лимит 200 строк (системный)
`/api/transactions` отдаёт по умолчанию **limit=200** (`ORDER BY date DESC`).
Любая выборка «за весь год/всё» БЕЗ `.limit(...)` обрезается → пропадают ранние
месяцы. Уже починено в Сводке и Операциях (`.limit(100000)`). **Где будет «дай
всё» — ставить высокий limit.** Это была причина «пропавшей коммуналки» в Сводке.

## 3. Характер категорий (ось планирования)

Характер задаётся на уровне **подкатегории** (у одной категории подкатегории
могут быть разного характера; пример: «Автомобиль»: Бензин — Повседневный,
ТО — Эпизодический). Три значения:

| Характер | Что | В Бюджете |
|---|---|---|
| **Повседневный** | регулярные переменные траты | Конверты |
| **Фиксированный** | регулярные обязательные (стабильная сумма) | Правила |
| **Эпизодический** | разовые/нерегулярные | События |

Доходы тоже имеют характер: зарплата (основная работа) — Фиксированный; остальное
(подарки, подработка, кешбек) — Эпизодический. `expense_type` (`fixed`/`variable`)
выводится из характера (fixed ⟺ Фиксированный).

## 6. Бюджет — механика правил/конвертов/событий

- **Правило** (Фиксированный): `plan_rules`, разворачивается (материализуется) в
  таблицу `plan` (`source='plan_rule'`, `source_rule_id`). Материализация —
  `plan_materialize.materialize_rules` (current+next месяц при правке любого
  правила + крон). Кредиты материализуются отдельно (`source='loan_schedule'`,
  `/api/loans/generate-plan`).
- **Конверт** (Повседневный): `category_budgets` (per category+year+month). На
  Бюджете факт vs бюджет + прогноз. Suggest сумм: адаптивное окно (все полные
  месяцы, cap 6) + trimmed mean (≥3 мес → среднее без максимума).
- **Событие** (Эпизодический): разовая plan-строка на месяц (`source='manual'`,
  account_to='Расход'). **Эпизодический ДОХОД** вносится в блоке «Доходы»
  (account_from='Доход', разовая plan-строка, не правило). Дефолт-дата события =
  отображаемый месяц.

### Область правки/удаления правила: «этот месяц / все будущие» (durable)
- `plan.pinned` (bool) — ручная правка месяца; материализация её **не сносит**.
- `plan_rule_skips` (rule×год×месяц) — «удалить только этот месяц».
- **Правка «этот месяц»** → `POST /api/plan-rules/{id}/set-month` (создаёт/заменяет
  pinned-строку, работает даже в будущем месяце, где правило ещё не развёрнуто).
- **Правка «все будущие»** → `PATCH /api/plan-rules/{id}?clear_from_year&month`
  (меняет правило + снимает ручные правки/пропуски с текущего месяца и далее).
- **Удалить этот месяц** → `POST /api/plan-rules/{id}/skip-month`. **Удалить все
  будущие** → `DELETE /api/plan-rules/{id}` (CASCADE).
- В UI: тоггл «Изменить все будущие» в модалке; меню ⋮ правила — два «Удалить».
- Строка правила показывает **эффективную сумму месяца** (`ruleEffAmt` =
  материализованная строка месяца или дефолт правила); KPI Доходы/Фиксированные
  тоже считают по `ruleEffAmt`. Пилюли плановых строк и статус «Исполнено/Ожидается»
  убраны.

## 9. Метрики и формулы (текущее состояние)

- **Safe-to-Spend (STS, «Свободно»):** общий модуль `metrics_core.safe_to_spend`
  (тот же код для `/api/metrics` и крона). Компоненты: `b0` (операц. баланс),
  `i_remain` (плановый доход до конца месяца), `f_remain` (фикс. платежи +
  обязательства до конца месяца), `v_remain` (поведенческий остаток повседневных),
  пополнение резерва/подушки. STS может быть отрицательным (структурный дефицит).
- **DSR (Кредитная нагрузка):** `monthly_payments / ГАРАНТИРОВАННЫЙ доход`, где
  гарантированный = сумма дохода по категориям `character='Фиксированный'`
  (`metrics_core.monthly_fixed_income_sum`). Если фикс-доход не размечен — фолбэк к
  общему доходу. В ответе: `dsr.income_base`, `dsr.income_kind` (`fixed`/`total`).
  Пороги: <30% green, ≤45% yellow, иначе red. Карточка подписывает «от фикс./общего
  дохода».
- **«Расходы по категориям» на Обзоре:** второе число — **план из правил**, а не
  поведенческий прогноз. План категории = Σ материализованных плановых строк за
  месяц (правила + график кредита), **оба плеча кредита** (тело→Обязательства +
  процент→Расход), считается по (категория, характер). Если плана нет — фолбэк к
  поведенческому (повседневные) или факту. Подпись «факт → план».
- **Классификация в Операциях (`opFlow`)** — КРИТИЧНО, был тонкий баг:
  ```
  isLoanCat(t)  = категория/подкатегория содержит «Кредит»
  isDebtAcc(a)  = a ∈ {Обязательства, Дебиторка}
  opFlow(t):
    from==='Доход'                       → income
    to==='Расход'                        → isLoanCat ? obligation : expense
    isDebtAcc(from) || isDebtAcc(to)     → obligation
    иначе                                → transfer
  ```
  Порядок важен: **`to==='Расход'` проверяется РАНЬШЕ счетов** — иначе расход со
  счёта `Обязательства` (купил в долг, `Обязательства→Расход`) ошибочно станет
  долгом «получено» (+зелёный). Кредитный **процент** (Расход + кат. «Кредиты») →
  Долги; **тело** (→Обязательства) → Долги; обычный расход → Расходы. Так Долги =
  тело+процент (сходится с Бюджетом).
- **KPI «Расходы · план» в Операциях** включает **бюджет конвертов**
  (`category_budgets` за период) — иначе план занижен (повседневные не в `plan`).
- **Баланс счёта (общая формула для всех):** `initial_balance + Σ(amount where
  account_to=имя) − Σ(amount where account_from=имя)`. Пассив хранится
  отрицательным; долг = `abs(balance)`. Операция `Обязательства→Расход` уменьшает
  баланс Обязательства (растёт долг) и одновременно расход (Дт Расход / Кт
  Обязательства). Классификация `opFlow` на баланс НЕ влияет.
- **Снапшоты прогноза:** `forecast_cron.py` пишет `forecast_snapshots` для каждого
  активного юзера ежедневно 23:55 (systemd `horizon-forecast.timer`, Persistent).
  Подтверждено: работает (status 0), 1 active user.

## 10. Критичные моменты / gotchas

- **Middleware (`middleware.py`):** защищает ТОЛЬКО `/api/*` (кроме OAuth-флоу).
  Для 401 **возвращать `JSONResponse`, НЕ `raise HTTPException`** — внутри
  `BaseHTTPMiddleware` исключение отдаётся как 500. (Дважды ловили инцидент с
  «Internal Server Error».)
- **Владелец таблиц для миграций:** `ALTER TABLE` требует владельца `horizon`.
  Уже сделан `ALTER TABLE … OWNER TO horizon` для `category_budgets`, `goals`,
  `plan_rules`, `loans`. Если новая миграция ALTER’ит таблицу и падает «must be
  owner» — сменить владельца и доприменить.
- **`plan` и `categories` создаются вне миграций** (исходная схема); миграции их
  только ALTER’ят (с `IF NOT EXISTS`).
- Деплой добавления нового backend-файла — в `deploy.yml` И `deploy_remote.sh` (§1).
- Миграции прогоняются на КАЖДОМ деплое; делать идемпотентными.

## 11. P&L-прототип Сводки

- Стоял вопрос: превратить Сводку в **личный P&L** (отчёт о прибылях/убытках).
  Решение: **да, стоит** — данные уже всё содержат (характеры, разделение
  кредита).
- **Модель P&L (ключевое решение):**
  ```
   Доходы (фикс + эпизод)
  − Операционные расходы (Фиксированные / Повседневные / Эпизодические)
  = Операционный профит
  − Проценты по кредитам
  = ЧИСТЫЙ ПРОФИТ
  ── под чертой (распределение, на профит НЕ влияет) ──
  → Погашение тела долга
  → В сбережения / резервы
  ```
  Тело кредита и сбережения — **под чертой** (движения по балансу, не расходы).
  Над чертой из кредита — только **проценты**.
- **Прототип готов:** `horizon-summary-pnl.html` (корень, standalone, демо-данные
  2025/2026, корректная математика, бренд-стиль, тоггл гранулярности, столбцы
  месяцы + Среднее + Год, без ограничения ширины). Не интегрирован в приложение.
- Открытые вопросы прототипа: строка «Изменение свободных средств»; заёмы отдельной
  строкой vs нетто; План/Факт (пока только Факт).
